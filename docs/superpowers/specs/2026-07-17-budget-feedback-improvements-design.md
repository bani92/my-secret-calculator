# 가계부 사용성 피드백 개선 설계

## 목표

며칠 사용하면서 확인된 월 수입, 전월 이월, 수입 추가, 최근 지출 정렬 및 수정 흐름을 개선한다. 사용자는 PC와 모바일에서 같은 방식으로 월 예산을 관리할 수 있어야 하며, 특히 모바일 화면에서 버튼과 팝업이 깨지거나 겹치지 않아야 한다.

## 선택한 방향

기존 `LedgerTab.vue` 중심의 입력 흐름을 유지하되, 월 수입 영역에 두 개의 보조 행동을 추가한다.

- `저장`: input에 적힌 금액으로 현재 월 수입을 덮어쓴다.
- `전월 이월`: 팝업에서 전월 남은 돈과 반영 후 금액을 확인한 뒤 현재 월 수입에 더한다.
- `수입 추가`: 팝업에서 추가 금액을 입력하고 반영 후 금액을 확인한 뒤 현재 월 수입에 더한다.
- 최근 지출은 `createdAt` 기준 최신순으로 보여준다.
- 최근 지출 항목에는 `수정` 버튼을 추가하고, 날짜, 분류, 금액, 메모 수정은 팝업에서 처리한다.

저장 아래에 입력 영역이 펼쳐지는 방식은 이번 범위에서 사용하지 않는다. 수입 관련 보조 행동은 현재 월 수입을 바로 바꾸므로, 팝업으로 변경 전후 금액을 확인하게 하는 편이 실수 방지에 적합하다.

## 월 수입 영역 UX

월 수입 영역은 다음 구조를 사용한다.

```text
월 수입                         [전월 이월] [수입 추가]

이번 달 총 수입
[ 2,800,000 ]                  [저장]
```

모바일에서는 버튼들이 우측에 고정되지 않고 자연스럽게 줄바꿈된다.

```text
월 수입
2026-07

[전월 이월] [수입 추가]

이번 달 총 수입
[ 2,800,000 ]
[저장]
```

`saveIncome()`은 저장 후 input을 비우지 않는다. 저장에 성공하면 저장된 금액이 포맷된 상태로 input에 남아 있어야 한다.

## 전월 이월 동작

`전월 이월` 버튼을 누르면 선택 월의 바로 전월을 계산한다. 예를 들어 현재 선택 월이 `2026-07`이면 전월은 `2026-06`이다.

전월의 남은 돈은 기존 월 요약 계산과 같은 기준을 사용한다.

```text
전월 남은 돈 = 전월 수입 - 전월 지출 합계
```

전월 남은 돈이 0원보다 크면 확인 팝업을 표시한다.

```text
전월 남은 돈을 이월할까요?

전월: 2026-06
전월 남은 돈: 185,000원
현재 월 수입: 2,800,000원
반영 후 월 수입: 2,985,000원

[취소] [추가]
```

`추가`를 누르면 현재 월 수입에 전월 남은 돈을 더해 저장한다.

전월 남은 돈이 0원 이하이면 수입을 변경하지 않고 `이월한 남은 돈이 없습니다` 안내를 표시한다. 이 안내는 같은 화면 흐름 안에서 확인 가능한 메시지로 제공하며, 사용자가 잘못된 상태로 저장했다고 느끼지 않게 한다.

## 수입 추가 동작

`수입 추가` 버튼을 누르면 추가 금액 입력 팝업을 표시한다.

```text
수입 추가

추가 금액
[ 300,000 ]

현재 월 수입: 2,800,000원
반영 후 월 수입: 3,100,000원

[취소] [추가]
```

추가 금액은 0보다 커야 한다. 0원, 빈 값, 숫자로 해석할 수 없는 값은 저장하지 않고 팝업 안에 안내한다.

`추가`를 누르면 현재 월 수입에 입력 금액을 더해 저장한다. 이 동작은 수입 input에 직접 적고 `저장`을 누르는 덮어쓰기 동작과 다르다.

## 최근 지출 정렬

최근 지출은 지출 날짜가 아니라 생성 시각 기준으로 보여준다.

- 기준 필드: `createdAt`
- 정렬: `createdAt` 내림차순
- 보조 기준: `createdAt`이 비어 있는 오래된 데이터가 있을 경우 `date` 내림차순을 fallback으로 사용한다.

사용자는 날짜를 과거로 입력하더라도 방금 추가한 지출이 최근 지출 상단에 보여야 한다.

## 최근 지출 수정

최근 지출 항목은 `수정`과 `삭제` 버튼을 가진다.

```text
점심 김치찌개
2026-07-17 · 식비
9,000원                  [수정] [삭제]
```

`수정`을 누르면 팝업을 표시한다.

```text
지출 수정

날짜      [2026-07-17]
분류      [식비]
금액      [9,000]
메모      [점심 김치찌개]

[취소] [저장]
```

저장 시 날짜, 분류, 금액, 메모를 수정할 수 있다. 날짜가 변경되면 `month`도 새 날짜의 `YYYY-MM` 값으로 변경한다. 생성 시각인 `createdAt`은 수정해도 유지한다.

금액은 0보다 커야 하며, 메모는 기존 추가 흐름과 같이 앞뒤 공백을 제거한다.

## 데이터 모델과 저장소

`Expense` 타입에는 `createdAt`을 포함한다. 사용자가 데이터 확인 결과 모든 지출 데이터에 `createdAt`이 존재한다고 확인했으므로, Supabase 로딩 시 `created_at`을 `createdAt`으로 매핑한다.

로컬 저장소와 가져오기/내보내기 호환성을 위해 다음 방식을 사용한다.

- 새로 추가되는 지출은 앱에서 `createdAt`을 생성한다.
- Supabase에서 로드되는 지출은 `created_at`을 사용한다.
- 테스트 또는 오래된 JSON에 `createdAt`이 없으면 `date` 기반 fallback을 사용한다.
- 지출 수정은 `updateExpense(expense)` 저장소 메서드를 추가한다.

## 오류 처리

- 월 수입 저장, 전월 이월, 수입 추가, 지출 수정은 저장 실패 시 화면 상태를 성공한 것처럼 바꾸지 않는다.
- 저장 실패 메시지는 기존 앱의 일반 저장 실패 메시지 흐름을 따른다.
- 전월 이월 금액이 없을 때는 오류가 아니라 안내 메시지로 처리한다.
- 팝업 입력값이 잘못된 경우 팝업을 닫지 않고 해당 팝업 안에 안내한다.

## 모바일 UI 요구사항

모바일에서 다음을 반드시 확인한다.

- 월 수입 영역의 `전월 이월`, `수입 추가`, `저장` 버튼이 겹치지 않는다.
- 긴 금액이 input, 버튼, 요약 카드 밖으로 튀어나가지 않는다.
- 팝업은 작은 화면에서 좌우 여백을 유지하고 세로로 자연스럽게 배치된다.
- 지출 항목의 메모가 길어도 금액과 버튼 영역을 밀어 깨뜨리지 않는다.
- 수정 팝업의 날짜, 분류, 금액, 메모 필드는 1열로 쌓인다.
- 팝업 하단의 `취소`, `추가`, `저장` 버튼은 터치하기 충분한 크기를 유지한다.

검증 대상 화면 폭은 최소 360px, 일반 모바일 390px, 데스크톱 1280px로 한다.

## 테스트

TDD로 다음 동작을 검증한다.

- `monthExpenses`가 `createdAt` 기준 최신순으로 정렬된다.
- `createdAt`이 없는 데이터는 `date` 기준 fallback으로 정렬된다.
- 월 수입 저장 후 input 값이 비워지지 않는다.
- 전월 남은 돈이 0원 이하이면 `이월한 남은 돈이 없습니다`를 표시하고 저장하지 않는다.
- 전월 남은 돈이 있으면 현재 월 수입에 더해 저장한다.
- `수입 추가`에서 입력한 금액이 현재 월 수입에 더해 저장된다.
- 잘못된 추가 금액은 저장하지 않는다.
- 지출 수정은 날짜, 분류, 금액, 메모를 저장하고 날짜 변경 시 `month`를 갱신한다.
- 지출 수정 저장 실패 시 기존 화면 상태가 유지된다.

## 이번 범위에서 제외

- 수입 추가 또는 전월 이월 내역을 별도 히스토리로 저장하는 기능
- 지출 수정 이력 관리
- 삭제 확인 팝업 변경
- 실시간 동기화
- Spring Boot 백엔드 추가

## Self-Review

- Placeholder scan: 미정 항목이나 임시 문구는 없다.
- Internal consistency: `저장`은 덮어쓰기, `전월 이월`과 `수입 추가`는 더하기로 일관된다.
- Scope check: 월 수입과 최근 지출 피드백 개선에 한정되어 있으며 이력 관리와 실시간 동기화는 제외했다.
- Ambiguity check: 팝업 방식, 버튼명, 전월 잔액 0원 이하 안내, 모바일 검증 폭, 저장 실패 기준을 명시했다.
